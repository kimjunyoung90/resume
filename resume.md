## 전자세금계산서 서비스 운영 및 유지보수 (2021.09 ~ 현재)

- 월 6,500만 건, 최대 1,200 TPS 트래픽을 처리하는 세금계산서 발행·신고 및 전자문서 플랫폼 운영

### 대량 지연 대응 및 병목 해소

#### 문제
- 국세청 신고 기간 중 특정 API p95 응답시간이 25초까지 상승
- DB/서버 리소스는 정상 → 애플리케이션 내부 병목 의심

#### 해결 전략
- Thread Dump 분석으로 특정 메서드의 synchronized 블록에서 BLOCKED 스레드 대기 현상 발견
- 단기적으로 Scale-out 적용, 근본적으로 synchronized 제거 및 내부 로직 재구조화
- ExecutorService 기반 동시성 검증 + JMeter 부하 테스트 수행

#### 성과
- 급격한 트래픽 증가에 신속히 대응하여 서비스 안정적인 운영 유지
- Thread Dump 분석을 통해 병목 구간을 정확히 식별하고 원인 제거
- TPS 3 → 25(8배 향상)
- p95 응답시간 25초 → 1.5초 단축

### 분산 트랜잭션 정합성 확보 (Saga Orchestration)

#### 문제
- 포인트 차감 → 세금계산서 발행 프로세스에서 발행 실패시 포인트 복구가 누락되는 정합성 문제 발생

#### 해결 전략
- Orchestration 기반 Saga 패턴 설계 및 구현
- 포인트 차감 → 발행 → 실패 시 보상 트랜잭션 수행 프로세스 구현
- 연계 서비스가 포인트 하나로 단순해, 프로세스 제어가 용이한 Orchestration 방식을 적용
- 보상 트랜잭션 실패 시 로그 적재하여 수기처리 방안 마련

#### 성과
- 데이터 정합성 문제 근본 해결
- 시스템 신뢰도 향상

### 무료 발행 이벤트 동시성 제어

#### 문제
- 신규 채권 발행 서비스 출시 프로모션으로 고객별 무료 발행 100건 이벤트 진행
- 무료 발행 건수 차감 시 SELECT 후 UPDATE로 인한 동시성 문제 발생

#### 해결 전략
- 경합 확률이 낮은 구조를 고려해 Optimistic Lock + Spring Retry 적용
- update_timestamp 컬럼을 버전 컬럼으로 활용해 충돌 감지
- 재시도 3회, 랜덤 Backoff 딜레이로 재충돌 방지

#### 성과
- 발행 동시 요청 상황에서도 중복 차감 이슈 없이 안정적으로 처리
- Optimistic Lock 기반의 효율적 동시성 제어 구조 구현

## 뉴스 데이터 수집 시스템 개선 (2024.03)

- 여러 언론사의 기사 데이터를 수집·가공·조회하는 서비스 구조 개선

### 구조적 문제 해결

#### 문제
- 특정 언론사에 맞춰 설계된 DB 구조로 인해 신규 언론사 추가 시 확장성 제한
- 기사 수집 및 가공 로직이 언론사별로 중복되어 신규 제휴 확장 시 개발 공수가 과도하게 증가

#### 해결 전략

1. **플랫폼 표준 카테고리 기반 DB 구조 재설계**
    - 기존에는 언론사별 카테고리를 정의하는 `categories` 만 존재, 해당 카테고리를 기준으로 서비스가 운영 되고 있었음
    - 정치·경제·사회·IT·문화 등 플랫폼 전용 카테고리를 정의한 `platform_categories` 테이블 신설
    - 언론사별 카테고리를 플랫폼 표준 카테고리에 매핑하는 `category_mapping` 테이블 설계
    - 신규 언론사 추가 시 DB 스키마 변경 없이 매핑 데이터만 등록하면 확장 가능
    - 플랫폼 표준 카테고리 구조 변경 시에도 매핑 데이터 수정만으로 서비스 전반에 즉시 반영되도록 설계
2. **조회 시 동적 매핑 구조 적용**
    - 카테고리 조회 시점에 `category_mapping`을 참조하여 플랫폼 표준 카테고리 기준으로 통합 조회
    - 카테고리 정보는 자주 변경되지 않지만 지속적으로 데이터베이스를 조회해야하는 문제 발생
    - 실시간성을 요구하는 데이터는 아니라 Redis 기반 캐시 계층을 도입해 매핑 데이터 조회 부하를 제거
3. **전략패턴 기반 기사 수집 프로세스 공통화**
    - 언론사별 데이터 수집 부분을 전략 패턴을 사용하여 공통 NewsCollector 인터페이스로 통합
    - Service Layer에서 파라미터를 통해 런타임에 언론사 타입에 맞는 전략을 선택해 실행하도록 설계
    - 신규 언론사 연동 시 기존 코드를 수정하지 않고 전략 클래스 추가만으로 확장 가능

#### 성과

- 신규 언론사 연동 및 표준 카테고리 변경 시 DB 구조 수정 없이 데이터만으로 즉시 반영 가능
- 수집·조회 로직 공통화로 신규 제휴 추가 시 개발 공수 절감
- Redis 캐시 적용으로 조회 시 JOIN 부하 제거 및 응답 속도 개선
