# Projects

## 전자세금계산서 서비스 운영 및 유지보수 (2021.09 ~ 현재)

- **월 6,500만 건, 최대 1,200 TPS** 트래픽을 처리하는 세금계산서 발행·신고 및 전자문서 플랫폼 운영

### API 대량 지연 개선

#### 상황
- 트래픽 집중 기간(월 1~10일) 요청량 급증으로 문서 발행 API 지연 발생
- APM에서 **latency 증가(p95 25초)**, **throughput 감소** 현상 관측
- DB 및 외부 호출 구간은 정상 응답 → **애플리케이션 내부 처리 병목**으로 판단

#### 행동
**1) Thread 상태 분석을 통한 원인 특정**
- **jstack 기반 Thread Dump 분석** 과정에서 **BLOCKED 스레드 대량 포착**
- Stack Trace를 통해 Blocking이 발생하는 메서드 확인
- 특정 메서드가 **`synchronized`에 의해 직렬 처리**되고 있음을 발견
- 코드 확인 시 synchronized가 보호하려던 공유 자원이 없고 멀티 인스턴스 환경에서는 실효성이 없어 키워드 제거

**2) 부하 테스트를 통한 개선 효과 검증**
- staging 환경 부재로 개발 환경에서 **부하 테스트 수행(JMeter)**
- **Stepping Thread Group**을 사용하여 동시 요청 수를 단계적으로 증가(20→60)
- 제거 전·후 latency와 TPS를 동일 조건에서 비교

**3) 운영 적용 및 모니터링**
- 개선사항 적용 후 Elastic APM과 Thread 상태를 지속 관찰
- 월초 피크 구간에서도 병목이 재발하지 않음

#### 결과
- latency **2.4초 → 1.8초(약 24% 개선)**
- TPS **13.5 → 19.8(약 47% 증가)**
- APM 상 월초 피크 구간에서도 latency 지표가 안정적으로 유지

### API 멱등성 개선

#### 상황
- 고객사의 비정상 요청(중복 요청·응답 미수신)으로 동일 세금계산서가 간헐적으로 **중복 발행**
- 기존 구조: 고객 제공 멱등키를 세금계산서 테이블에 멱등키 컬럼을 두어 관리했으나, **NULL 허용·UNIQUE 제약 없음**
- 중복 방지 로직이 **애플리케이션 레벨 SELECT 체크**에만 의존 → 동시 요청 시 둘 다 "없음"으로 판단해 **중복 발행 가능**

#### 행동
**1) 멱등키 전용 테이블 신규 설계**
- **멱등키만 관리하는 테이블을 신규 구축**
- 기존 500GB 세금계산서 테이블에 UNIQUE 인덱스 추가 시 테이블 락으로 인한 서비스 중단 위험 고려
- 신규 건부터 멱등키를 새 테이블에 함께 적재, 기존 세금계산서 테이블은 그대로 유지

**2) 복합 Primary Key로 DB 레벨 중복 차단**
- PK 구성: `(고객 고유번호, 멱등키 값)`으로 **고객 단위 멱등키 고유성 보장**
- 동시 요청이 들어와도 먼저 INSERT된 요청만 성공하고, 나머지는 **PK 충돌로 즉시 차단**


#### 결과
- 도입 후 **중복 발행 0건** (고객 문의 없음으로 확인)
- 서비스 중단 없이 **멱등성 보장 구조 개선**

### 분산 시스템간 트랜잭션 정합성 개선

#### 상황
- 세금계산서 발행 프로세스는 포인트 시스템과 연계되어 **포인트 차감 → 세금계산서 발행 → (실패 시) 포인트 환불** 순으로 진행
- 보상 트랜잭션(포인트 환불) 실패 시 데이터 정합성을 회복할 메커니즘이 부재
- 문서 발행 실패 시 포인트가 환불되지 않는 문제가 발생하여 고객 금전적 손실 야기

#### 행동
1) **보상 결과 영속화**
- **보상 트랜잭션 실패 이력**을 저장하는 전용 테이블 생성
- 문서 번호, 포인트 코드, 상태(READY, RETRY, SUCCESS, FAIL), 재처리 횟수, 사용자 고유 번호로 구성
- 보상 트랜잭션 실패 시 실패 이력 적재

2) **보상 트랜잭션 재처리 프로세스** 구축
- **Jenkins 스케줄을 활용**한 API 호출 방식으로 구현
- 처리 순서
  1. 미처리(`READY`) 보상 트랜잭션 조회
  2. 상태를 `RETRY`로 업데이트
  3. 환불 API 호출
  4. 성공 시 `SUCCESS`, 실패 시 재시도 횟수 증가 후 `READY`로 복구
- 최대 재시도 횟수(5회) 초과 시 `FAIL`로 업데이트, **자사 메신저 알림 발송** 및 백오피스 수동 복구 기능 구현

#### 결과
- 도입 후 포인트·문서 데이터 불일치 **0건**
- 보상 실패 건 자동 재처리로 수동 개입 없이 정합성 복구 가능한 체계 확보

### 동시성 문제 해결 (Optimistic Lock)

#### 상황
- 신규 채권 발행 서비스 출시 프로모션으로 고객별 무료 발행 100건 이벤트 진행
- 무료 발행 건수 차감 시 SELECT 후 UPDATE로 인한 동시성 문제 발생

#### 행동
1) **동시성 문제 재현 및 분석**
- **ExecutorService 기반 동시성 테스트 코드** 작성하여 이슈 재현
- 10개 스레드풀 생성만으로도 동시성 이슈 발생 확인
- 단일 쿼리로 개선 검토했으나, 히스토리를 쌓아야 해서 로직 개선 불가

2) **Optimistic Lock 적용**
- 한 회사당 100건, 회사에 일반적으로 채권 발행 담당자가 소수이고 초기 서비스라 사용자가 적어 경합이 적을 것으로 판단
- 경합 확률이 낮은 구조를 고려해 **Optimistic Lock + Spring Retry** 적용
- **version 컬럼을 추가하여 충돌 감지**

3) **재시도 전략**
- backoff, random, multiplier, maxDelay 설정 적용
- 경합 상황에서 모든 트랜잭션이 즉시 재시도를 수행하면 재경합 가능성이 높아 대기시간 지정(backoff)
- 재시도 시점을 랜덤하게 지정하여 동시 경합 가능성 완화(random)
- 재시도 횟수 증가 시 대기 시간을 점진적으로 증가시켜 경합 완화(multiplier)
- 임의로 재시도 설정 변경 시 지연이 무한정 길어지는 현상 방지(maxDelay)

4) **최종 실패 처리**
- 최종 재시도 실패 시 recover 메서드를 실행하여 실패 응답 반환 및 로그 적재
- 비관적 락으로의 전환 시점 파악

#### 결과
- 발행 동시 요청 상황에서도 **중복 차감 이슈 없이 안정적으로 처리**
- **Optimistic Lock 기반**의 효율적 동시성 제어 구조 구현

## 뉴스 데이터 수집 시스템 개선 (2024.03)

- 언론사의 기사 데이터를 수집·가공·조회하는 서비스 구조 개선

### DB 구조 재설계 및 확장성 개선

#### 상황
- 기존 뉴스 서비스는 단일 언론사만을 고려해 설계되어 `categories`에 언론사 고유 카테고리를 그대로 저장하고 UI에 노출하는 구조였음
- 이후 서비스 고도화 과정에서 다수 언론사의 데이터를 수집하고, 각 언론사가 제공하는 카테고리를 플랫폼 서비스 기준(예: 정치·IT 등)에 맞게 통합하여 제공해야 한다는 요구사항이 전달되었음

#### 행동
1) **N:M 매핑 구조**의 데이터베이스 재설계
- `platform_categories` 테이블 신설: 플랫폼(WEHAGO, NAHAGO)별 표준 카테고리 관리
- `categories` 테이블 재정의: 언론사별 원본 카테고리를 서브 카테고리로 활용
- `categories_mapping` 테이블 추가: **N:M 관계 구현**으로 하나의 언론사 카테고리가 여러 플랫폼 카테고리에 매핑 가능하도록 설계

2) 쿼리 복잡도 증가 문제 해결
- DB 구조 변경으로 기존 단순 쿼리 → JOIN으로 복잡도 증가
- **MyBatis SQL 스니펫**을 활용하여 공통 JOIN 로직 재사용

3) 기존 API 호환성 유지
- platform 추가 및 응답값 변경으로 기존 클라이언트 영향 발생
- **API 버전 관리(v1, v2)**로 점진적 마이그레이션 지원
- **3개월 병행 운영** 후 v1 단계적 제거

4) **전략 패턴** 기반 수집 로직 공통화
- 언론사마다 수집 Endpoint와 응답 데이터 구조가 상이함
- 전체 수집 흐름은 유지하면서 데이터 수집 및 가공 부분을 **전략 클래스로 분리**
- 신규 언론사 연동 시 전략 클래스만 추가하면 되어 기존 비즈니스 로직을 수정하지 않고 확장 가능한 구조로 설계(**개방 폐쇄 원칙** 준수)

#### 결과
- **N:M 매핑 구조**로 **DB 구조 변경 없이 매핑 데이터만 추가**하여 **비즈워치 외 2개 언론사 확장**
- **전략 패턴** 적용으로 신규 언론사 추가 시 **개발 공수 절감**
- **API 버전 관리**로 **기존 클라이언트 영향 없이** 신규 기능 배포

#### 상황
- 한국어·영어·일본어 3개 언어로 서비스를 제공하기 위해 클라이언트 코드 내 한국어를 다국어 도구(react-intl)가 읽을 수 있는 형태로 변환 작업의 필요
- 매번 개발 시마다 수행해야 하는 단순 반복 작업으로 빠르게는 몇 분, 길게는 며칠이 소요

#### 행동
- **babel parser**를 활용해 React 컴포넌트를 **AST(Abstract Syntax Tree)**로 파싱
- **JSX Element**와 **StringLiteral** 노드를 순회하며 한국어 패턴 **정규식으로 탐지**
- 하드코딩된 한국어 문자열을 **react-intl의 FormattedMessage 컴포넌트로 자동 변환**
- 탐지된 한국어 문자열을 **JSON 형태의 메시지 파일에 자동 추가**
- 기존 메시지 파일과 **병합하여 키 중복 및 누락 방지**
- **Commander.js**로 사용자 친화적인 **CLI 인터페이스** 구현

#### 결과
- 다국어 작업 시간 **5분 이내로 단축**
- **반복 업무가 감소**하여 개발 효율 증가
- 수동 작업을 코드 기반 자동화로 **다국어 변환 실수 및 누락 감소**
- **오픈소스 공개**로 개발 생태계에 기여
