# Projects

## 전자세금계산서 서비스 운영 및 유지보수 (2021.09 ~ 현재)

- **월 6,500만 건, 최대 1,200 TPS** 트래픽을 처리하는 세금계산서 발행·신고 및 전자문서 플랫폼 운영

### Latency 9초 -> 350ms, 처리량 47% 향상

#### 문제
- 월 초(1~10일) 전체 트래픽의 40%가 집중되는 패턴
- 해당 시기 특정 API 응답시간이 최대 **9초 이상**으로 치솟는 현상 발생

#### 해결 과정
**1) APM 분석**
- 모니터링 분석 시 애플리케이션 내부에서 소모되는 시간의 비중이 전체 소요 시간의 90% 이상으로 컸음
- 시스템의 CPU·메모리 확인 결과 CPU 5%, 메모리 30%로 정상 범위로 판단
- 애플리케이션 내부 스레드 비정상 동작 의심

**2) Thread 분석**
- **Thread Dump 분석** 과정에서 **BLOCKED 스레드 대량 포착**
- Stack Trace 확인 결과 특정 메서드가 **`synchronized`에 의해 직렬 처리**되고 있음을 발견
- 코드 레벨 분석 후 synchronized 제거

**3) synchronized 제거에 따른 영향도 분석**
- synchronized 제거 시 직렬 처리되던 요청이 병렬로 실행되면서 **DB 부하 증가** 우려
- **DB CPU**: 평균 10~20%, 최대 60%로 트래픽 병렬 처리시 안전하다고 판단
- **DB 커넥션**: 현재 커넥션 풀 크기(max 40 × 8대 = 320) 대비 DB 최대 한도(2,205)는 여유가 있어, 커넥션 부족 시 풀 크기 증가 고려

**4) 부하 테스트를 통한 개선 효과 검증**
- 제거 전·후 latency와 TPS를 동일 조건에서 비교(개발 환경)
- TPS **13.5 → 19.8(약 47% 증가)**

#### 성과
- 월초 피크 구간 병목 현상 제거
- 운영 기준 latency 최대 **9초 이상 → 350ms** 안정적 유지

### 인덱스 + 쿼리 튜닝으로 조회 성능 개선(6,800ms → 0.121ms)

#### 문제
- 국세청 매입 세금계산서 조회 API에서 지연 발생
- APM 분석 결과 **쿼리 실행이 전체 응답시간의 99% 이상** 차지하여 쿼리 병목으로 확인
- 대상 테이블은 **국세청 수집 데이터 테이블**로 당시 **5,300만 건** 규모

#### 해결 과정
**1) 실행계획 분석**
- cost **2,513,296** / 실행시간 **6,800ms**
- Index Scan을 사용함에도 비용이 높아 쿼리·인덱스 점검 필요하다고 판단

**2) 쿼리 분석**
- 쿼리 조건 중 **매입자 사업자번호**: 등치 조건
- 쿼리 조건 중 **작성일자(YYYYMMDD 문자열)**: 범위 조건, 단 `to_timestamp` 함수로 감싸여 있음
- 컬럼에 함수 적용 시 인덱스 원본 값과 달라져 인덱스 활용 불가 → **첫 번째 원인으로 식별**

**3) 인덱스 분석**
- 테이블에 생성된 인덱스 확인 결과 **작성일자** 포함된 인덱스 부재
- 매입자 사업자번호는 **특정 복합 인덱스의 4번째(비선행) 위치**였음
- PostgreSQL이 **인덱스 전체를 순회하는 형태로 동작**
- 두 조건을 효율적으로 탐색할 **복합 인덱스 부재** → **두 번째 원인으로 식별**

**4) 통합 해결책 적용**
- **`to_timestamp` 함수 제거**: YYYYMMDD 포맷은 사전순과 날짜순이 동일해 문자열 비교로 동일 결과 보장
- **(매입자 사업자번호, 작성일자) 복합 인덱스 생성**
  - 컬럼 순서: 등치 조건(매입자 사업자번호) 선행, 범위 조건(작성일자) 후행 — 등치 조건으로 데이터 범위를 먼저 좁힌 후 범위 탐색하는 것이 효율적이라고 판단
  - 국세청 수집 데이터는 **적재 빈도 대비 조회 빈도가 훨씬 높은 읽기 우세 테이블**이라 인덱스 추가에 따른 쓰기 비용 영향 미미하다고 판단


#### 성과
| 항목 | 개선 전 | 개선 후 |
|---|---|---|
| 옵티마이저 추정 cost | 2,513,296 | 8.59 |
| 실측 실행시간 | 6,800ms | 0.121ms |
| 작성일자 처리 | Filter | Index Cond |

- 인덱스 크기 701MB로 운영 부담 미미한 수준

### MSA 데이터 정합성 개선

#### 문제
- 담당 서비스는 Orchestration 방식의 MSA 환경
- 클라이언트 요청 -> 세금계산서(Orchestration) -> 포인트 차감 -> 세금계산서 발행 -> 포인트 차감 취소 순으로 진행
- 문서 생성 실패 시 포인트가 취소 되지 않는 문제 발생 -> 고객 금전적 손실 야기
- 보상 트랜잭션(포인트 환불) 실패 시 데이터 정합성을 회복할 메커니즘이 부재

#### 해결 과정
1) **보상 결과 영속화**
- **보상 트랜잭션 실패 이력**을 저장하는 전용 테이블 생성
- 문서 번호, 재처리 상태, 재처리 횟수, 사용자 고유 번호로 구성
- 보상 트랜잭션 실패 시 실패 이력 적재

2) **보상 트랜잭션 재처리 프로세스** 구축
- **Jenkins 스케줄을 활용**한 API 호출 방식으로 구현
- 처리 순서
  1. 미처리 실패건 조회
  2. 환불 API 호출
  3. 성공 시 `SUCCESS`, 실패 시 재시도 횟수 증가
- 최대 재시도 횟수(5회) 초과 시 `FAIL`로 업데이트, **자사 메신저 알림 발송** 및 백오피스 수동 복구 기능 구현

3) **`Lock`을 활용한 스케줄 중복 실행 방지**
- 현재 스케줄 방식에서 이전 스케줄이 완료되기 전에 다음 스케줄이 실행될 수 있는 구조
- 단일 스케줄 작업 서버라 `ReentrantLock`으로 방지

#### 성과
- 도입 후 포인트·문서 데이터 불일치 **0건**
- 보상 실패 건 자동 재처리로 수동 개입 없이 정합성 복구 가능한 체계 확보

### 사내 금전 손실 방지

#### 문제
- 신규 채권 발행 서비스 출시 프로모션으로 회사별 무료 채권 100건 발행 이벤트 진행
- 카운트 기반 쿠폰으로 무료 발행 건수 차감 시 SELECT 후 UPDATE로 인한 동시성 문제 발생

#### 해결 과정
1) **동시성 문제 재현 및 분석**
- **ExecutorService, CountDownLatch 기반 동시성 이슈** 재현
- Race Condition에 따른 중복 차감 발생 확인

2) **Optimistic Lock 적용**
- 회사별 채권 발행 담당자가 소수이며 초기 서비스로 사용자가 적어 **경합 빈도가 낮은 환경**으로 판단
- 락을 잡지 않고 충돌 시 재시도하는 **Optimistic Lock**을 선택

3) **재시도 + 트랜잭션 설계**
- 재시도 포함 **하나의 트랜잭션**으로 구성하여 원자성 보장
- 트랜잭션 내부에서 버전 충돌 예외 발생 시 트랜잭션이 롤백되어 재시도 자체가 무의미해지는 문제 존재
- **noRollbackFor** 속성으로 버전 충돌 예외 시 롤백을 방지하여 재시도가 정상 동작하도록 구현

#### 성과
- 발행 동시 요청 상황에서도 **중복 차감 이슈 없이 안정적으로 처리**
- **Optimistic Lock 기반**의 효율적 동시성 제어 적용

## 뉴스 데이터 수집 시스템 개선 (2024.03)

- 언론사의 기사 데이터를 수집·가공·조회하는 서비스 구조 개선

### DB 구조 재설계 및 확장성 개선

#### 문제
- 기존 뉴스 서비스는 단일 언론사만을 고려해 설계되었고, 신문사에서 제공하는 카테고리 정보를 `paper_categories`에 저장하고 뉴스 데이터는 카테고리에 매핑되는 1:N 구조 였음
- 이후 서비스 고도화 과정에서 다수 언론사의 데이터를 수집하고, 각 언론사가 제공하는 뉴스 데이터를 플랫폼 서비스 기준(예: 정치·IT 등)에 맞게 통합하여 제공해야 한다는 요구사항이 전달되었음

#### 해결 과정
1) **N:M 매핑 구조**의 데이터베이스 재설계
- `platform_categories` 테이블 신설: 자사 서비스 플랫폼(WEHAGO, NAHAGO)별 표준 카테고리 관리
- `categories_mapping` 테이블 추가: **N:M 관계 구현**으로 하나의 언론사 카테고리가 여러 플랫폼 카테고리에 매핑 가능하도록 설계

2) **전략 패턴** 기반 수집 로직 공통화
- 언론사마다 수집 API와 응답 데이터 구조가 상이한 문제 발생
- 전략 패턴을 도입하여 전체 수집 흐름은 유지하면서 데이터 수집 및 가공 부분을 **전략 클래스로 분리**
- 신규 언론사 추가 시 전략 클래스만 추가하여 확장 가능한 구조로 설계(**개방 폐쇄 원칙** 준수)

#### 성과
- **N:M 매핑 구조**로 **데이터 추가만으로** **기존 외 2개 언론사 추가 확장**
- **전략 패턴** 적용으로 신규 언론사 추가 시 **확장 구조 설계**

### 상품 조회 Redis 캐싱

#### 문제
- 상품 조회는 커머스에서 전체 트래픽의 대부분을 차지하지만 상품 정보의 변경 빈도는 낮음 → 매번 DB 조회는 자원 낭비
- Redis 기반 캐싱 적용으로 DB 부하를 감소시키고 응답 속도를 개선

#### 해결 과정
**1) 캐싱 및 캐시 갱신 전략**
- 캐시 Hit: 캐싱 데이터 반환, 캐시 Miss: DB 조회 후 데이터 캐싱 후 반환
- 캐시 갱신 전략: Lost Update 방지를 위한 수정 트랜잭션 commit 완료 후 TransactionEventListener로 삭제
- 삭제 실패 시 TTL 동안 일시적 불일치 허용

**2) 상품 다건 조회 캐싱**
- 상품 다건 조회 시 자주 조회되는 1페이지 상품만 캐싱하여 캐시 저장소 효율적 사용
- 이중 캐시 구조로(상품 목록 id + 상세 정보) 캐싱 데이터 갱신 시 목록 캐시 전체 갱신 방지
- 캐시 갱신으로 인한 정렬 변경 시 TTL 동안 일시적 불일치 허용

**3) Cache Stampede 방지**
- 만료된 캐시 조회가 집중적으로 발생할 때 트래픽이 DB로 한번에 몰리지 않도록 분산 락 기반 stampede 현상 방지
- 다건의 상품 정보 일괄 캐싱 시 jitter를 활용하여 TTL 일괄 만료 방지
- 미스 -> 락대기 -> 캐시 재조회 -> DB 조회 흐름

**4) Redis 장애 격리**
- Redis 장애가 서비스 장애로 확산되지 않도록 예외 처리를 통해 정상 흐름 유지


#### 성과
- 상품 조회 트래픽을 캐시 레이어로 흡수해 DB 부하 감소 및 응답 속도 개선
- 단건 + 목록 ID 이중 캐시 구조로 **단건 갱신만으로 목록 응답까지 반영**, 캐시 메모리 사용량 절감
- 분산 락 + TTL Jitter로 Cache Stampede 방지, Redis 장애 시 DB 응답으로 서비스 연속성 확보

### Kafka 기반 이벤트 처리

#### 문제
- 사용자 구매 상품 정보 집계의 필요로 인한 Kafka 적용
- 메시지 발행은 외부 시스템이라 트랜잭션으로 묶이지 않아 이벤트 유실 및 유령 이벤트 발생 가능
- 메시지 중복 발행 시 컨슈머의 중복처리로 인한 데이터 신뢰성 저하
- 메시지 발행・소비 실패 시 재시도 프로세스로 인해 후속 메시지 처리 정체의 위험성

#### 해결 과정
**1) Outbox 패턴 적용**
- 비즈니스 로직과 발행할 이벤트를 동일한 트랜잭션으로 묶고 발행할 메시지를 outbox 테이블에 적재
- 스케쥴러로 outbox를 읽어 메시지 발행, 발행 완료시 완료 처리
- 발행 실패 시 retryCount 증가 → 최대 재시도 횟 수 초과 시 DLQ로 관리 → 운영자가 수동 분석·복구

**2) 메시지 소비 멱등 구조 설계**
- 메시지 소비 시 비즈니스 로직과 동일한 트랜잭션으로 묶어 소비된 메시지를 `event_handled`에 저장(`eventId`)
- 중복 메시지 수신 시 `eventId`를 DB 레벨에서 유니크 제약으로 멱등성 보장
- 지속된 소비 실패 시 소비 불가능한 메시지 DLT로 관리

#### 성과
- 비즈니스 트랜잭션과 Outbox 적재를 묶어 **이벤트 유실·유령 이벤트 동시 차단**, Kafka 장애 시에도 비즈니스 흐름 보장
- `eventId` + DB 유니크 제약 이중 안전망으로 at-least-once 환경에서 컨슈머 중복 처리 차단
- 재시도 한계 초과 메시지는 DLQ/DLT로 격리해 정상 메시지 흐름 유지, 운영자 수동 분석·복구 경로 확보
