# Projects

## 전자세금계산서 서비스 운영 및 유지보수 (2021.09 ~ 현재)

- **월 6,500만 건, 최대 1,200 TPS** 트래픽을 처리하는 세금계산서 발행·신고 및 전자문서 플랫폼 운영

### Latency 9초 -> 300ms, 처리량 N배 개선

#### 문제
- 월 초(1~10일) 전체 트래픽의 40%가 집중되는 패턴
- 해당 시기 특정 API 응답시간이 최대 **9초 이상**으로 치솟는 현상 발생

#### 해결 과정
**1) 모니터링 확인**
- 모니터링 분석 시 애플리케이션 내부에서 소모되는 시간의 비중이 전체 소요 시간의 90% 이상으로 컸음
- 시스템의 CPU·메모리 확인 결과 CPU 5%, 메모리 30%로 정상 범주로 판단
- 애플리케이션 내부 스레드 비정상 동작 의심

**2) Thread 분석**
- **Thread Dump 분석** 과정에서 **BLOCKED 스레드 대량 포착**
- Stack Trace 확인 결과 특정 메서드가 **`synchronized`에 의해 직렬 처리**되고 있음을 발견
- 코드 확인 시 synchronized가 보호하려던 공유 자원이 없고, 멀티 인스턴스 환경에서는 **동일 JVM 내에서만 직렬화를 보장**하므로 실효성이 없어 키워드 제거

**2) synchronized 제거에 따른 영향도 분석**
- synchronized 제거 시 직렬 처리되던 요청이 병렬로 실행되면서 **DB 부하 증가** 우려
- **DB CPU**: 평균 10~20%, 최대 60%로 트래픽 병렬 처리시 안전한다고 판단
- **DB 커넥션**: 현재 커넥션 풀 크기(max 40 × 8대 = 320) 대비 DB 최대 한도(2,205)는 여유가 있어, 커넥션 부족 시 풀 크기 증가 고려

**3) 부하 테스트를 통한 개선 효과 검증**
- staging 환경 부재로 개발 환경에서 **부하 테스트 수행(JMeter)**
- 제거 전·후 latency와 TPS를 동일 조건에서 비교
- TPS **13.5 → 19.8(약 47% 증가)**

**4) 운영 적용 및 모니터링**
- 개선사항 적용 후 APM과 DB 상태를 지속 관찰
- 월초 피크 구간에서도 병목이 재발하지 않음

#### 성과
- 월초 피크 구간 병목 현상 제거
- latency 최대 **9초 이상 → 350ms** 안정적 유지

### 데이터 중복 적재 문제 해결

#### 문제
- 고객사의 비정상 요청(중복 요청·응답 미수신)으로 동일 세금계산서가 간헐적으로 **중복 발행**

#### 해결 과정
**1) 원인 분석
- 멱등키를 세금계산서 테이블에 저장하고 있었으나, **UNIQUE 제약조건 부재**로 중복된 멱등키 가능
- 중복 방지 로직이 **데이터 조회 후 유무 체크**에 의존하여 동시 요청 시 **중복 발행 가능**

**2) 제약조건 검토**
- Unique 제약조건 추가를 먼저 검토했으나, 해당 컬럼에 이미 일반 인덱스가 존재하여 **DROP 후 재생성 필요** → 삭제 시점부터 재생성 완료까지 인덱스 부재로 **서비스 장애 우려**
- 서비스 무중단 + 최소 변경이 요구되는 상황이라 **멱등키 전용 테이블 신규 생성** 선택

**3) 멱등키 전용 테이블 신규 설계**
- **멱등키만 별도로 관리하는 신규 테이블 도입**
- 신규 건부터 멱등키를 새 테이블에 함께 적재

**4) 복합 PK로 DB 레벨 중복 차단**
- PK 구성: `(고객 고유번호, 멱등키 값)`
- 고객사마다 멱등키 중복이 가능하기 때문에 **고객번호를 포함한 복합 PK**로 고객 단위 멱등성 보장
- 동시 요청이 들어와도 먼저 INSERT된 요청만 성공하고, 나머지는 **PK 충돌로 즉시 차단**

#### 성과
- 도입 후 **중복 발행 0건**
- 서비스 중단 없이 **멱등성 보장 구조 개선**

### MSA 데이터 정합성 개선

#### 문제
- 담당 서비스는 Orchestration 방식의 MSA 환경
- 세금계산서 생성 프로세스는 포인트 시스템과 연계되어 **포인트 차감 → 세금계산서 생성 → (실패 시) 포인트 환불** 순으로 진행
- 문서 생성 실패 시 포인트가 환불되지 않는 사례가 발생 -> 고객 금전적 손실 야기
- 보상 트랜잭션(포인트 환불) 실패 시 발생한 문제. 데이터 정합성을 회복할 메커니즘이 부재

#### 해결 과정
1) **보상 결과 영속화**
- **보상 트랜잭션 실패 이력**을 저장하는 전용 테이블 생성
- 문서 번호, 재처리 상태, 재처리 횟수, 사용자 고유 번호로 구성
- 보상 트랜잭션 실패 시 실패 이력 적재

2) **보상 트랜잭션 재처리 프로세스** 구축
- **Jenkins 스케줄을 활용**한 API 호출 방식으로 구현
- 처리 순서
  1. 미처리 실패건 조회
  2. 환불 API 호출
  3. 성공 시 `SUCCESS`, 실패 시 재시도 횟수 증가 후 `READY`로 복구
- 최대 재시도 횟수(5회) 초과 시 `FAIL`로 업데이트, **자사 메신저 알림 발송** 및 백오피스 수동 복구 기능 구현

3) **`Lock`을 활용한 스케줄 중복 실행 방지**
- 현재 스케쥴 방식에서 이전 스케줄이 완료되기 전에 다음 스케줄이 실행될 수 있는 구조
- 단일 스케쥴 작업 서버라 `ReentrantLock`으로 방지

#### 성과
- 도입 후 포인트·문서 데이터 불일치 **0건**
- 보상 실패 건 자동 재처리로 수동 개입 없이 정합성 복구 가능한 체계 확보

### 사내 금전적 손실 방지

#### 문제
- 신규 채권 발행 서비스 출시 프로모션으로 고객별 무료 채권 100건 발행 이벤트 진행
- 카운트 기반 쿠폰으로 무료 발행 건수 차감 시 SELECT 후 UPDATE로 인한 동시성 문제 발생

#### 해결 과정
1) **동시성 문제 재현 및 분석**
- **ExecutorService, CountDownLatch 기반 동시성 테스트 코드** 작성
- Race Condition에 따른 동시성 이슈 발생 확인

2) **구조 개선 검토**
- 단일 쿼리로의 수정 검토
- 무료 건수 이력 적재로 인해 불가

3) **낙관적 락 적용**
- 한 회사당 100건, 회사에 일반적으로 채권 발행 담당자가 소수이고 초기 서비스라 사용자가 적어 경합이 적을 것으로 판단
- 경합이 낮은 상황을 고려해 **Optimistic Lock + 재시도** 적용

4) **트랜잭션 조정**
- 재시도 포함 하나의 트랜잭션으로 구성하여 **원자성 보장**
- **noRollbackFor** 속성을 사용하여 버전 충돌 시 롤백 방지 구현

#### 성과
- 발행 동시 요청 상황에서도 **중복 차감 이슈 없이 안정적으로 처리**
- **Optimistic Lock 기반**의 효율적 동시성 제어 구조 구현

## 뉴스 데이터 수집 시스템 개선 (2024.03)

- 언론사의 기사 데이터를 수집·가공·조회하는 서비스 구조 개선

### DB 구조 재설계 및 확장성 개선

#### 문제
- 기존 뉴스 서비스는 단일 언론사만을 고려해 설계되어 `categories`에 언론사 고유 카테고리를 그대로 저장하고 UI에 노출하는 구조였음
- 이후 서비스 고도화 과정에서 다수 언론사의 데이터를 수집하고, 각 언론사가 제공하는 카테고리를 플랫폼 서비스 기준(예: 정치·IT 등)에 맞게 통합하여 제공해야 한다는 요구사항이 전달되었음

#### 해결 과정
1) **N:M 매핑 구조**의 데이터베이스 재설계
- `platform_categories` 테이블 신설: 플랫폼(WEHAGO, NAHAGO)별 표준 카테고리 관리
- `categories` 테이블 재정의: 언론사별 원본 카테고리를 서브 카테고리로 활용
- `categories_mapping` 테이블 추가: **N:M 관계 구현**으로 하나의 언론사 카테고리가 여러 플랫폼 카테고리에 매핑 가능하도록 설계

2) 쿼리 복잡도 증가 문제 해결
- DB 구조 변경으로 기존 단순 쿼리 → JOIN으로 복잡도 증가
- MyBatis SQL 재사용 문법을 활용하여 공통 JOIN 로직 재사용

3) **전략 패턴** 기반 수집 로직 공통화
- 언론사마다 수집 API와 응답 데이터 구조가 상이한 문제 발생
- 전략 패턴을 도입하여 전체 수집 흐름은 유지하면서 데이터 수집 및 가공 부분을 **전략 클래스로 분리**
- 신규 언론사 추가 시 전략 클래스만 추가하여 확장 가능한 구조로 설계(**개방 폐쇄 원칙** 준수)

#### 성과
- **N:M 매핑 구조**로 **데이터만 추가만으로** **기존 외 2개 언론사 추가 확장**
- **전략 패턴** 적용으로 신규 언론사 추가 시 **개발 공수 절감**

### API 명세서 작성 자동화

#### 문제

- 신규 서비스를 개발하는 동료가 API 명세서 작성에 시간이 많이 소요된다는 고충을 공유
- 명세서 작성에 낭비되는 시간으로 인한 개발 비용의 증가

#### 해결 과정
- **AI Agent(Claude Code, Codex)**를 사내 Wiki에 연동하는 방법 탐색

- AI가 코드를 탐색해 명세서를 **자동으로 작성**하는 동작 확인
- 팀원 및 부서 개발자들이 바로 사용할 수 있도록 **연동 가이드 문서 및 프롬프트 공유**

#### 성과
- 반복적인 API 명세 작성 작업이 **자동화**되어 낭비되는 비용을 줄임
- 주변 업무의 비효율성을 쉽게 개선할 수 있다는 영향 전파
