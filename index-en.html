Electronic Tax Invoice Service Operations & Maintenance 2021.09 ~ Present

  • Operated a tax-invoice issuance/filing and electronic-document platform handling 65 million transactions per month, up to 1,200 TPS
Latency 9s → 350ms, Throughput +47%
Problem
  • 40% of total monthly traffic is concentrated in the first 10 days of the month
  • During that period, the response time of a specific API spiked to over 9 seconds
Approach

1) APM analysis

  • Monitoring showed that time spent inside the application accounted for over 90% of the total elapsed time
  • System CPU was 5% and memory 30%, both within normal range
  • Suspected abnormal behavior of threads inside the application

2) Thread analysis

  • Detected a large number of BLOCKED threads during thread dump analysis
  • The stack trace revealed that a specific method was being serialized by synchronized
  • Removed synchronized after code-level analysis

3) Impact analysis of removing synchronized

  • Removing synchronized would let previously serialized requests run in parallel, raising concerns about increased DB load
  • DB CPU: 10–20% on average, up to 60%, judged safe for parallel traffic
  • DB connections: current pool size (max 40 × 8 instances = 320) had headroom against the DB max limit (2,205); planned to increase pool size if connections ran short

4) Validating the improvement with load testing

  • Compared latency and TPS before/after removal under identical conditions (dev environment)
  • TPS 13.5 → 19.8 (about +47%)
Result
  • Eliminated the bottleneck during the early-month peak
  • Production latency stayed stable, improving from over 9s to 350ms
Query Performance Improved via Indexing + Query Tuning (6,800ms → 9.412ms)
Problem
  • Latency occurred in the NTS (National Tax Service) purchase tax-invoice lookup API
  • APM analysis showed query execution accounted for over 99% of the total response time
  • Target table: NTS-collected data, around 53 million rows at the time
Approach

1) Execution plan analysis

  • cost 2,513,296 / execution time 6,800ms
  • Cost was high despite an Index Scan, so the query and index needed review

2) Query analysis

  • Business registration number: equality condition
  • Issue date (YYYYMMDD string): range condition, but with a to_timestamp function applied
  • Applying the function changes the value from the indexed original, making the index unusable → index review needed

3) Index analysis

  • No index including the issue date existed → removing the function alone could not solve it
  • The business registration number sat in the 4th (non-leading) position of a composite index
  • No composite index to efficiently scan both conditions

4) Query tuning and index creation

  • Removed to_timestamp: since YYYYMMDD is lexicographically ordered = date-ordered, string comparison guarantees the same result
  • Created a composite index on (buyer business registration number, issue date)
  • Column order: equality (business number) leading + range (issue date) trailing
Result
Metric Before After
Optimizer estimated cost 2,513,296 5,306.11
Measured execution time 6,800ms 9.412ms
MSA Data Consistency Improvement
Problem
  • The service I owned ran in an orchestration-based MSA environment
  • Flow: client request → tax invoice (orchestration) → point deduction → invoice issuance → point deduction rollback
  • When document creation failed, points were not rolled back → causing financial loss to customers
  • No mechanism existed to restore data consistency when the compensating transaction (point refund) itself failed
Approach

1) Persisting compensation results

  • Created a dedicated table to store the history of failed compensating transactions
  • Composed of document number, reprocessing status, retry count, and user ID
  • Recorded a failure entry whenever a compensating transaction failed

2) Building a compensating-transaction reprocessing process

  • Implemented as API calls driven by a Jenkins schedule
  • Processing order
    • Query unprocessed failures
    • Call the refund API
    • On success mark SUCCESS, on failure increment the retry count
  • When the max retry count (5) is exceeded, update to FAIL, send an internal messenger alert, and provide a back-office manual recovery feature
Result
  • Zero point/document data mismatches after rollout
  • Established a system that restores consistency without manual intervention by automatically reprocessing failed compensations
Preventing Internal Financial Loss
Problem
  • A launch promotion for a new bond-issuance service ran an event granting each company 100 free bond issuances
  • Deducting the free-issuance count via a count-based coupon caused a concurrency issue from SELECT-then-UPDATE
Approach

1) Reproducing and analyzing the concurrency issue

  • Reproduced the concurrency issue using ExecutorService and CountDownLatch
  • Confirmed duplicate deductions caused by a race condition

2) Applying an optimistic lock

  • Judged it a low-contention environment since few people per company issue bonds and the service was new with few users
  • Chose an optimistic lock that retries on conflict instead of holding a lock

3) Retry + transaction design

  • Composed retries within a single transaction to guarantee atomicity
  • A version-conflict exception inside the transaction would roll it back, making retries meaningless
  • Used the noRollbackFor attribute to prevent rollback on version-conflict exceptions so retries work correctly
Result
  • Handled concurrent issuance requests reliably with no duplicate-deduction issues
  • Applied efficient concurrency control based on an optimistic lock

Product Lookup Caching with Redis
Problem
  • Product lookups make up most of commerce traffic, yet product info changes infrequently → querying the DB every time wastes resources
  • Applied Redis-based caching to reduce DB load and improve response time
Approach

1) Caching and cache-invalidation strategy

  • Cache hit: return cached data; cache miss: query the DB, then cache and return
  • Invalidation strategy: to prevent lost updates, delete via a TransactionEventListener after the update transaction commits
  • On delete failure, allow temporary inconsistency for the TTL duration

2) Caching multi-item product lookups

  • Cached only the frequently viewed first page of products to use cache storage efficiently
  • A two-tier cache (product list IDs + detail info) avoids refreshing the entire list cache when one item changes
  • For ordering changes caused by cache refresh, allow temporary inconsistency for the TTL duration

3) Preventing cache stampede

  • Used a distributed lock to prevent traffic from hitting the DB all at once when expired-cache lookups surge
  • Applied jitter when bulk-caching many products to avoid TTLs expiring all at once
  • Flow: miss → wait for lock → re-check cache → query DB

4) Isolating Redis failures

  • Used exception handling to keep the normal flow so a Redis failure does not cascade into a service outage
Result
  • Absorbed product-lookup traffic at the cache layer, reducing DB load and improving response time
  • The single-item + list-ID two-tier cache lets a single-item update propagate to list responses, cutting cache memory usage
  • Distributed lock + TTL jitter prevent cache stampede; on Redis failure, DB responses ensure service continuity
Kafka-based Event Processing
Problem
  • Adopted Kafka to aggregate users' purchased-product information
  • Message publishing is an external system not bound to the transaction, allowing event loss and ghost events
  • Duplicate message publishing degrades data reliability through duplicate consumer processing
  • On publish/consume failure, the retry process risks stalling subsequent message processing
Approach

1) Applying the Outbox pattern

  • Bound business logic and the event to publish in the same transaction, writing the message to an outbox table
  • A scheduler reads the outbox, publishes messages, and marks them done on success
  • On publish failure, increment retryCount → on exceeding the max retries, move to a DLQ → operator manually analyzes and recovers

2) Designing idempotent message consumption

  • On consumption, bind to the same transaction as the business logic and store the consumed message in event_handled (eventId)
  • On receiving a duplicate message, guarantee idempotency via a DB-level unique constraint on eventId
  • On persistent consumption failure, route unconsumable messages to a DLT
Result
  • Binding the business transaction and outbox write blocks both event loss and ghost events, ensuring the business flow even during a Kafka outage
  • eventId + a DB unique constraint form a double safety net that blocks duplicate consumer processing in an at-least-once environment
  • Messages exceeding the retry limit are isolated to a DLQ/DLT to keep the normal message flow, with a manual analysis/recovery path for operators
