전자세금계산서 서비스 운영 및 유지보수 2021.09 ~ 현재

  • B2B, B2C 전자세금계산서 발행·신고 및 기타 전자문서 관리 플랫폼
  • 월 평균 6,500만 건, 최대 1,200 TPS의 대규모 트래픽 처리 시스템
1) 대량 지연 병목 해소
Problem
  • 세금계산서 신고 기간 중 트래픽이 몰리면서 p95 응답시간이 25초까지 상승
  • DB, 서버 리소스 이상 없음 → 애플리케이션 처리량 한계 의심
Solution
  • Thread Dump 분석으로 특정 구간에서 BLOCKED 상태의 스레드 다수 확인
  • 특정 메서드에 synchronized 키워드로 인한 락 경합이 원인임을 파악
  • 단기적으로 Scale-out, 근본 해결로 synchronized 제거
  • ExecutorService 기반 동시성 검증 및 JMeter 부하 테스트 수행
Result
  • TPS 3 → 25 (8배 개선), p95 응답시간 15초 → 1.5초 단축
  • Thread 상태 분석으로 애플리케이션 병목 정확히 진단
2) 분산 트랜잭션 정합성 확보 (Saga Orchestration)
Problem
  • 포인트 차감 → 세금계산서 발행 시 분산 시스템 간 트랜잭션 오류 발생
  • 발행 실패 시 포인트 복구 누락으로 데이터 정합성 오류 발생
Solution
  • Orchestration 기반 Saga 패턴 설계 및 구현
  • 전자세금계산서 시스템이 Orchestrator 역할 수행
  • 포인트 차감 → 발행 → 실패 시 보상 트랜잭션 수행
  • Kafka 등 이벤트 브로커 고려했으나, 단일 서비스 내 트랜잭션 제어를 위해 Orchestration 방식 채택
Result
  • 데이터 정합성 및 트랜잭션 복원력 확보
3) 이벤트 동시성 제어
Problem
  • 신규 채권 발행 서비스 출시 이벤트로 무료 발행 100건 제공
  • 무료 발행 건수 차감 시 SELECT 후 UPDATE로 인한 동시성 문제 발생
Solution
  • 경합 확률이 낮은 구조를 고려해 Optimistic Lock + Spring Retry 적용
  • update_timestamp 컬럼을 버전 컬럼으로 활용해 충돌 감지
  • 재시도 3회, 랜덤 Backoff 딜레이로 재충돌 방지
Result
  • 이벤트 트래픽 100만 건 이상에서도 중복 차감 0건
  • 분산락 없이 경량화된 동시성 제어 달성

Result

의료마이데이터 신규 서비스 개발 2024.05 ~ 2024.07

  • 공공기관에 분산된 개인 의료 데이터를 수집하여 통합 관리하는 건강정보 플랫폼 개발
  • 건강검진, 진료기록, 처방기록 등의 정보를 조회·관리 할 수 있는 시스템 구축
외부 시스템 장애 격리 구현
Problem
  • 공공기관으로 부터 수집한 개인 의료 데이터는 내부 데이터 수집 시스템에서 관리
  • 의료마이데이터 시스템은 플랫폼 전체적인 역할을 하며 데이터 수집 시스템에서 관리하는 데이터를 조회 해야하는 구조
  • 데이터 수집 시스템 API 지연·장애 발생 시 의료마이데이터 서비스에 영향 우려
Solution
  • Hystrix Circuit Breaker 적용으로 외부 API 지연·장애로부터 서비스 보호
    • 도메인별(진료, 검진, 의료기록 등) 스레드풀 격리 및 큐 용량 제한(10)
    • 실패율 50% 이상 시 회로 차단(Open), 10초 후 Half-Open 전환
    • DynamicPropertyFactory 기반으로 설정값 30초 단위 동적 반영
  • 회로 차단 시 대체 응답("일시적 조회 불가") 반환
  • Resilience4j 대신 Spring 4.3 기술 스택과 맞는 Hystrix 선택
Result
  • 외부 API 지연 시에도 서비스 가용성 99.9% 유지
  • 톰캣 워커 스레드 고갈 방지 및 장애 전파 차단

뉴스 데이터 수집 시스템 개선 2024.03

  • 제휴 신문사에서 뉴스 데이터를 수집하여 플랫폼 사용자에게 뉴스 컨텐츠 제공
  • 뉴스 서비스는 초기 설계상 특정 1개 신문사 데이터만 수집·제공 → 신규 매체 확장이 어려운 구조
  • 다수 신문사 데이터를 수집·검색·제공할 수 있는 확장형 구조로 개선
구조적 문제 해결
Problem
  • 특정 1개 신문사만 고려하여 설계된 데이터베이스
  • 기사 수집·가공 로직이 중복되어 신규 매체 추가 시 개발 공수 증가
  • 외부 이미지 URL 직접 의존으로 원본 장애 시 이미지 미노출
Solution
  • 카테고리 표준화 및 DB 재설계
    • 플랫폼 전용 카테고리 테이블(정치·경제·사회·IT·문화 등) 설계
    • 신문사별 카테고리를 플랫폼 카테고리에 매핑하는 category_mapping 테이블 설계
    • 신규 매체 추가 시 매핑만으로 확장 가능
  • 수집·가공 모듈 공통화
    • 기사 데이터를 표준 DTO로 변환 후 UPSERT 정책 적용
    • 중복 적재 방지 및 최신 데이터 자동 갱신
  • 이미지 안정화
    • 수집 시점에 이미지 다운로드 후 NAS 저장, 내부 URL로 치환
    • 원본 서버 장애 시에도 안정적인 이미지 제공
Result
  • 신규 매체(전자신문) 연동 완료, 플랫폼 확장성 확보
  • 신규 매체 추가 시 DB 변경 불필요, 개발 공수 절감
  • 이미지 노출 실패율 1% 미만, 운영 효율성 및 유지보수성 향상
